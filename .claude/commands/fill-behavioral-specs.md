# Fill Behavioral Specs

You are tasked with filling in behavioral specifications (contracts, invariants) for a project that uses **spec-checker** — a structural and behavioral specification checker.

The project already has skeleton `.spec.yaml` files generated by `spec-checker init`. These skeletons contain structural information (types, function signatures, type constraints, dependencies) but are missing behavioral specifications. Your job is to read the source code and fill in the behavioral parts.

## Arguments

$ARGUMENTS — Optional: a source file path (e.g., `src/bridge.rs`) to process a single file. If omitted, process the entire project in dependency order.

## What to fill in

For each **function** in `exposes`:
- `requires`: Preconditions that must hold before calling the function. Examples: "amount > 0", "connection is open", "input is not empty", "caller has admin role"
- `ensures`: Postconditions guaranteed after the function returns. Examples: "returns sorted list", "balance increases by amount", "emits DepositEvent", "file handle is closed on error"
- `modifies`: State that the function changes. Examples: "self.balance", "global registry", "database connection pool"

For the **module**:
- `invariants`: Properties that always hold for this module. Examples: "withdrawn[token] <= deposited[token]", "no panic in public API", "no unsafe code", "thread-safe: all public methods are Send + Sync", "all errors are logged before propagation"

For **protocols** (if the module has lifecycle or stateful patterns):
- Refine `protocol.states`, `protocol.transitions`, and `protocol.balanced_pairs` if the auto-detected ones are incomplete

## Quality guidelines

Write specs that are:
1. **Verifiable** — An LLM or static checker can determine if the code satisfies them by reading the source. Avoid vague statements like "handles errors correctly". Prefer "returns Err on invalid input, never panics".
2. **Non-trivial** — Don't restate what the type signature already says. If a function takes `amount: u64`, don't write `requires: amount is a u64`. Instead write meaningful domain constraints.
3. **Behavioral** — Focus on what the code *does*, not what it *is*. Type structure is already captured in `type_constraints`.
4. **Concise** — Each spec should be one clear sentence or expression. No paragraphs.
5. **Consistent** — Use the same terminology across related modules. If module A calls it "connection", don't call it "link" in module B.

## Spec YAML format reference

```yaml
module: example
source_path: src/example.rs
language: rust

exposes:
  my_function:
    kind: function
    type_constraints:
      - "equals(param(input), String)"     # Already filled by init
      - "equals(return, Result)"            # Already filled by init
    requires:
      - input is not empty                  # YOU FILL THIS
      - connection is in ready state        # YOU FILL THIS
    ensures:
      - returns Ok with parsed result on valid input    # YOU FILL THIS
      - returns Err with descriptive message on failure # YOU FILL THIS
    modifies:
      - self.cache                          # YOU FILL THIS

  MyStruct:
    kind: type
    type_constraints:
      - "is_product(Self)"                  # Already filled
      - "has_field(Self, name)"             # Already filled
    ensures:
      - name is never empty after construction   # YOU FILL THIS (type invariant)

invariants:
  - "All public functions return Result, never panic"     # YOU FILL THIS
  - "No unsafe code"                                      # YOU FILL THIS
  - "Cache entries expire after TTL"                       # YOU FILL THIS

protocol:
  states: [uninitialized, ready, closed]
  initial: uninitialized
  terminal: [closed]
  transitions:
    - { from: uninitialized, call: new, to: ready }
    - { from: ready, call: process, to: ready }
    - { from: ready, call: close, to: closed }
  balanced_pairs:
    - ["open", "close"]
```

## Workflow

### Step 1: Get processing order

Run `spec-checker toposort ./specs` to get the dependency-ordered list of source files. This ensures you process leaf dependencies first, building up context.

If a single file was provided as argument, only process that file.

### Step 2: For each file (in order)

1. **Read the source file** to understand what the code does
2. **Read its existing spec** (the `.spec.yaml` file) to see what's already there
3. **Read the specs of its dependencies** (already filled from earlier iterations) for context on what contracts the dependencies guarantee
4. **Fill in the behavioral fields** — requires, ensures, modifies, invariants
5. **Write the updated spec** back to the `.spec.yaml` file
6. **Report progress** — tell the user which file you just completed

### Step 3: Validate

After processing all files, run:
```bash
spec-checker check .
```

This runs structural checks (without LLM) to catch any format errors or broken references in the specs you wrote. Fix any errors reported.

## Important rules

- **Never modify `type_constraints`** — these are structural and already correct from `init`
- **Never remove existing fields** — only add to `requires`, `ensures`, `modifies`, `invariants`
- **Preserve YAML formatting** — keep the existing structure, comments, and field order
- **Read before writing** — always read the source code before writing specs for it
- **Be conservative** — only write specs you're confident about from reading the code. It's better to have fewer accurate specs than many speculative ones.
- **Use domain language** — match the terminology used in the code's comments, variable names, and documentation
