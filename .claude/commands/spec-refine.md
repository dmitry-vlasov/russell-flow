# Spec Refine

You are tasked with refining existing specifications by adding detailed behavioral contracts for a project that uses **spec-checker**.

The project already has `.spec.yaml` files generated by `spec-checker init [--ai]`. These contain structural information (module description, API entities, layer, forbidden_deps, invariants). Your job is to **deepen** them with per-function behavioral contracts.

## Arguments

$ARGUMENTS — Optional: a source file path (e.g., `src/bridge.rs`) to process a single file. If omitted, process the entire project in dependency order.

## What to add

For each **function** in `exposes`:
- `requires`: Preconditions that must hold before calling. Examples: "amount > 0", "connection is open", "input is not empty"
- `ensures`: Postconditions guaranteed after return. Examples: "returns sorted list", "balance increases by amount", "emits DepositEvent"
- `modifies`: State that the function changes. Examples: "self.balance", "global registry"

For each **type** in `exposes`:
- `ensures`: Type invariants. Examples: "name is never empty after construction", "balance >= 0"

Refine existing fields if they can be made more precise:
- Sharpen `description` if vague
- Add missing `invariants` the init missed
- Refine `protocol` if auto-detected transitions are incomplete

## Quality guidelines

Write specs that are:
1. **Verifiable** — A checker can determine if code satisfies them. Prefer "returns Err on invalid input, never panics" over "handles errors correctly".
2. **Non-trivial** — Don't restate type signatures. Write meaningful domain constraints.
3. **Behavioral** — Focus on what the code *does*, not what it *is*.
4. **Concise** — One clear sentence or expression per spec.
5. **Consistent** — Use the same terminology across related modules.

## Workflow

### Step 1: Get processing order

Run `spec-checker toposort ./specs` to get dependency-ordered list. Process leaf dependencies first.

If a single file was provided as argument, only process that file.

### Step 2: For each file (in order)

1. **Read the source file** to understand what the code does
2. **Read its existing spec** to see what `init --ai` already generated
3. **Read dependency specs** (already refined) for context on upstream contracts
4. **Add behavioral contracts** — requires, ensures, modifies for each function
5. **Write the updated spec** back to the `.spec.yaml` file
6. **Update source_hash**:
   ```bash
   sha256sum <source-file> | cut -c1-16
   ```
   Set `source_hash: <that value>` in the spec YAML.
7. **Report progress** — tell the user which file you completed

### Step 3: Validate

After processing all files, run:
```bash
spec-checker check .
```
Fix any errors reported.

## Important rules

- **Always update `source_hash`** after modifying a spec
- **Never remove existing fields** — only add or refine
- **Preserve YAML formatting** — keep existing structure and field order
- **Read before writing** — always read source code before writing specs
- **Be conservative** — only write specs you're confident about. Fewer accurate specs beat many speculative ones.
- **Use domain language** — match terminology from the code
